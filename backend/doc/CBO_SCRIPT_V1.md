# Combo Builder Operations Script V1

Документ описывает текущую реализацию Combo Builder Operations Script для promo-акций `class=akciya`.

Короткое название:

- `CBO Script`

## Назначение

CBO Script V1 используется для исполняемых акций, которые:
- проверяют условие по составу заказа;
- начисляют баллы;
- дают скидку;
- добавляют подарок.

Если у `akciya` заполнены только:
- `id`
- `class=akciya`
- `name`
- `lore`

то такая акция считается информационной карточкой:
- она отображается в блоке акций;
- не требует `condition` и `reward`;
- не участвует в расчёте promo при заказе.

## Формат файла

Файл акции хранится в `backend/static/promo_items/.../item.txt`.

Формат строк:

```txt
key=value
```

Правила:
- одна строка = одно поле;
- пустые строки допустимы;
- строки, начинающиеся с `#`, считаются комментариями;
- повторять одно и то же поле нельзя;
- неизвестные поля запрещены.

## Поддерживаемые поля

### Обязательные для исполняемой акции на CBO Script

```txt
id=
class=akciya
name=
lore=
condition=
reward=
```

### Необязательные

```txt
notify=
reward_mode=
limit_per_order=
limit_per_user_per_day=
priority=
active=
start_at=
end_at=
```

### Значение полей

- `id`: числовой ID акции.
- `class`: для CBO Script V1 всегда `akciya`.
- `name`: название акции.
- `lore`: описание акции для пользователя.
- `condition`: условие срабатывания.
- `reward`: награда.
- `notify`: текст уведомления после применения акции.
- `reward_mode`: режим применения награды.
- `limit_per_order`: максимум срабатываний за один заказ.
- `limit_per_user_per_day`: максимум срабатываний на пользователя в день.
- `priority`: приоритет акции. Чем выше число, тем раньше акция обрабатывается.
- `active`: `true` или `false`.
- `start_at`: начало активности в ISO-формате.
- `end_at`: конец активности в ISO-формате.

## Пример рабочей акции

```txt
id=999
class=akciya
name=Тест FPV 74
lore=За 2 FPV-Друн начисляем 100 бонусов.
condition=ID(74).QTY >= 2
reward=POINTS(100)
notify=Начислены 100 бонусов
reward_mode=once
limit_per_order=1
limit_per_user_per_day=10
priority=100
active=true
```

## Пример информационной акции без CBO Script

```txt
id=2
class=akciya
name=Пройди опрос
lore=Описание акции для пользователя.
```

Такая акция:
- показывается как акция;
- не валится на валидации CBO Script;
- не применяется в `/payment` и `/payment/confirm`.

## Condition

### Операторы сравнения

Поддерживаются:

```txt
=
>
<
>=
<=
```

### Логические операторы

Поддерживаются только:

```txt
AND
OR
```

Операторы пишутся в верхнем регистре.

### Скобки

Поддерживаются круглые скобки для группировки:

```txt
( ID(74).QTY >= 2 AND ORDER.SUM >= 1000 )
```

### Метрики

#### 1. Конкретное блюдо по ID

```txt
ID(74).QTY
ID(74).SUM
```

Значения:
- `.QTY` = количество блюда в заказе;
- `.SUM` = сумма по этому блюду.

#### 2. Все блюда

```txt
ID.QTY
ID.SUM
```

Значения:
- `.QTY` = общее количество всех позиций;
- `.SUM` = сумма по всем позициям.

#### 3. Блюда по типу

```txt
ID.закуски.QTY
ID.закуски.SUM
ID.горячее.QTY
ID.алкоголь.QTY
```

Значение между `ID.` и `.QTY/.SUM` должно совпадать с типом блюда из меню.

#### 4. Группа конкретных ID

```txt
GROUP(10, 11, 12).QTY
GROUP(10, 11, 12).SUM
```

`GROUP()` не может быть пустым.

#### 5. Сумма заказа

```txt
ORDER.SUM
```

У `ORDER` поддерживается только `.SUM`.

### Примеры condition

```txt
condition=ID(74).QTY >= 2
condition=ID.закуски.QTY >= 2 AND ORDER.SUM >= 900
condition=GROUP(101, 205).QTY >= 2
condition=(ID(74).QTY >= 2 OR ID(75).QTY >= 2) AND ORDER.SUM >= 1000
```

## Reward

Поддерживаются 4 вида награды.

### 1. Начисление баллов

```txt
reward=POINTS(100)
```

Ограничение:
- число должно быть больше 0.

### 2. Скидка в процентах

```txt
reward=DISCOUNT_PERCENT(10)
```

Ограничение:
- значение от 1 до 100.

### 3. Скидка в рублях

```txt
reward=DISCOUNT_RUB(300)
```

Ограничение:
- значение больше 0.

### 4. Подарок

```txt
reward=GIFT(777, 1)
```

Формат:
- `GIFT(id_блюда, количество)`

Ограничения:
- item id должен существовать в меню;
- item должен быть активным;
- количество должно быть больше 0.

## reward_mode

Поддерживаются:

```txt
reward_mode=once
reward_mode=per_match
```

### once

Награда применяется один раз при выполнении условия.

### per_match

Награда применяется кратно количеству совпадений, но только если:
- условие состоит из одного сравнения;
- оператор в condition равен `>=`;
- порог положительный.

Пример:

```txt
condition=ID(101).QTY >= 2
reward=POINTS(50)
reward_mode=per_match
```

Если заказ содержит 6 единиц `ID(101)`, акция сработает 3 раза.

## limit_per_order

Ограничивает число срабатываний акции в рамках одного заказа.

Пример:

```txt
limit_per_order=1
```

## limit_per_user_per_day

Ограничивает число срабатываний акции для одного пользователя за день.

Пример:

```txt
limit_per_user_per_day=3
```

## active

Поддерживаются только:

```txt
active=true
active=false
```

## start_at и end_at

Формат: ISO timestamp.

Примеры:

```txt
start_at=2026-03-24T00:00:00
end_at=2026-03-31T23:59:59
```

Правило:
- `start_at` должен быть меньше либо равен `end_at`.

## Ошибки валидации

Типовые причины невалидности:

- нет `condition`;
- нет `reward`;
- пустой `notify`;
- неизвестное поле;
- дублирование поля;
- неверный оператор;
- неверная метрика;
- несуществующий `item_id`;
- несуществующий тип блюда;
- подарок ссылается на неактивное блюдо;
- `reward_mode=per_match` использован со сложным условием.

## Что не поддерживается в V1

Не поддерживается:

- `NONE`, `none` и другие заглушки CBO Script;
- отрицательные числа;
- дробные числа;
- операторы `!=`;
- reward без скобок;
- несколько `reward` в одной акции;
- несколько `condition` в одной акции;
- вложенные или составные reward.

## Практическое правило

Используйте:

- только `id/class/name/lore`, если нужна просто отображаемая акция без логики;
- полный CBO Script с `condition` и `reward`, если акция должна реально применяться к заказу.
