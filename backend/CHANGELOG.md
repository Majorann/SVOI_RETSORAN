### Promo, checkout и начисление баллов

- Добавлено отдельное DSL-ядро(Combo Builder Operations Script) акций в `backend/services/promotions/`:
  - AST, parser, validator, evaluator, applier, engine и checkout-helper для promo;
  - поддержка условий и наград вида `ID(...)`, `ID.<type>`, `ORDER.SUM`, `POINTS(...)`, `DISCOUNT_RUB(...)`, `DISCOUNT_PERCENT(...)`, `GIFT(...)`;
  - отдельная валидация reward mode, лимитов, окна активности и ссылок на существующие блюда/типы меню.
- Модель `PromoItem` расширена DSL-полями:
  - `condition`, `reward`, `notify`, `reward_mode`,
  - `limit_per_order`, `limit_per_user_per_day`,
  - `start_at`, `end_at`,
  - `dsl_valid`, `dsl_error`.
- Подтверждение оплаты больше не доверяет старому `preview`:
  - для `/payment/confirm` и delivery confirm позиции заказа заново собираются из актуального меню;
  - promo, скидки, `promo_points` и итоговые суммы теперь пересчитываются на сервере повторно перед записью заказа.
- `checkout preview token` теперь привязан к `user_id`, чтобы нельзя было подтвердить чужой или устаревший preview.
- В расчёте totals добавлена поддержка `discount_total`:
  - скидка вычитается до списания баллов;
  - `payable_total` и `bonus_earned` считаются уже от суммы после скидки и применения баллов.
- Для promo-акций выровнены правила времени и приоритета:
  - окна активности и дневные лимиты считаются в `APP_TIMEZONE`;
  - сортировка promo в UI и в runtime приведена к одному порядку по приоритету.
- Для checkout добавлен серверный promo preview:
  - отдельный route для пересчёта promo и скидок по текущей корзине;
  - frontend checkout/payment теперь использует пересчитанные promo-поля и уведомления.
- В `checkout` добавлен отдельный UI-блок применённой акции:
  - при соблюдении условий акции показывается название promo и её эффект;
  - блок оформлен в едином зелёном статусном стиле;
  - promo пересчитывается по текущей корзине до перехода на страницу оплаты.
- Страница `payment` приведена к тому же promo-представлению:
  - применённые акции, бонусные баллы и скидки показываются отдельным зелёным summary-блоком.
- В `Уведомлениях` DSL-акции снова отображаются как обычные promo-карточки без зелёного статуса применения:
  - зелёная подсветка оставлена только там, где акция реально применяется к заказу.
- Доработано отображение promo-summary в checkout и payment:
  - блок `Акция применена` больше не показывается, если нет реально применённых акций;
  - `notify` больше не выводится в checkout и payment;
  - `notify` остаётся только для пост-оплатных уведомлений.
- Для информационных акций `class=akciya` добавлен отдельный режим без исполняемого скрипта:
  - если в `item.txt` заполнены только `id`, `class`, `name`, `lore`, такая акция считается валидной для показа;
  - такие акции остаются в блоке акций и не попадают в runtime-расчёт promo.
- Шаблон `backend/static/promo_items/akciya/item.txt` упрощён до минимального формата информационной акции.
- Акция `Купи банку и пройди опрос` возвращена в `class=akciya` без фиктивного reward/condition:
  - отображается как акция;
  - не требует служебной заглушки в script-части;
  - не участвует в оплате как исполняемая promo.

### Promo admin и диагностика

- В admin service исправлена работа с promo без фотографии:
  - поиск папки акции теперь идёт по `class + id`, а не только по `photo`;
  - текстовые акции без изображения можно корректно обновлять и удалять.
- При загрузке promo контента добавлена серверная валидация DSL:
  - невалидные `akciya` скрываются из публичной выдачи;
  - причина ошибки сохраняется в `dsl_error` и отдаётся в admin/public metadata.
- В backend добавлена временная диагностика promo:
  - логирование загрузки promo-элементов;
  - логирование невалидного DSL у акций;
  - логирование применённых promo на этапах `/payment` и `/payment/confirm`.
- В admin panel доработан helper-конструктор DSL внутри существующего блока `Конструктор промо`:
  - helper работает только для `akciya`, для `reklama` он отключается;
  - добавлены builder-контролы для `condition` и `reward`;
  - текстовые поля DSL сохранены как основной источник истины и по-прежнему редактируются вручную.
- Для admin helper добавлена серверная кнопка `Проверить DSL` с явной цветовой индикацией результата:
  - успешная проверка отображается зелёным;
  - ошибка валидации отображается красным.
- UI helper в admin panel русифицирован для удобства:
  - переведены подписи полей, конструкторов условия и награды, а также служебные статусы проверки DSL.
- Подготовлена отдельная документация по синтаксису акций:
  - `backend/CBO_SCRIPT_V1.md`
  - пользовательское название DSL переименовано в `Combo Builder Operations Script` (`CBO Script`).

- Исправлено наложение dropdown `Ещё` над карточками меню:

- При открытии корзины на desktop убрано принудительное размытие меню:

### Postgres для акций и меню

- Акции `akciya` переведены на Postgres:
  - добавлены таблицы `promotions` и `promotion_applications`;
  - загрузка `akciya` идёт из БД, `reklama` остаётся файловой;
  - сохранение и удаление `akciya` в admin panel больше не пишет `item.txt`, а работает через Postgres.
- Дневные лимиты применения акций переведены на отдельную историю `promotion_applications`, а не на анализ прошлых заказов.
- Для `akciya` добавлен режим информационной акции без DSL:
  - можно сохранить только `name` и `lore`;
  - пустые `condition` / `reward` не валятся на валидации;
  - в admin panel в `reward_mode` добавлен вариант `нет`.
- Каталог блюд переведён на Postgres:
  - добавлена таблица `menu_items`;
  - публичное меню и admin menu читаются из БД;
  - сохранение блюда через `/admin/menu` пишет в Postgres;
  - изображения по-прежнему остаются в `backend/static/menu_items/...`, а в БД хранится путь.
- Для старых файлового меню и акций добавлена bootstrap-миграция в БД при первом запуске Postgres-режима.

### Автосогласование с хоста

- Добавлен startup-sync контента с хоста в БД:
  - при старте backend перечитывает файлы меню и акций;
  - выполняет `upsert` в `menu_items` и `promotions`;
  - записи, которых больше нет среди файлов хоста, автоматически удаляются из БД.
- В admin dashboard добавлена кнопка `Автосогласование`, запускающая тот же sync без перезапуска сайта.
- В audit log добавлено отдельное действие `content_autosync` с summary синхронизации.
